## Document Structure

### Section 1: Architecture Overview

Concise description of the full system for someone who hasn't read the code.

**Dataflow model** — Weight-stationary (TPU-style). Weights pre-loaded into MAC units and stay fixed. Activations stream left-to-right, partial sums accumulate top-to-bottom.

**MAC unit** (`rtl/mac_unit.sv`) — Parameterized pipeline (1/2/3 stages). Default 2-stage: stage 1 multiplies activation by stored weight, stage 2 adds incoming partial sum. Passthrough registers for activations (right) and weights (down, for loading chain).

**Systolic array** (`rtl/systolic_array.sv`) — NxN grid of MACs with activation skewing (row k delayed by k cycles). Weights loaded through top edge in reverse row order. Results drain from bottom edge with diagonal timing: C[m][j] at `drain_out[j]` at cycle m+j from first valid output.

**Controller FSM** (`rtl/controller.sv`) — Drives tiled matrix multiplication through phases per tile: LOAD_WEIGHTS → FEED → DRAIN → WRITEBACK → next tile or DONE. Handles full M/K/N tiling with read-modify-write accumulation for K-dimension tiles. Manages 1-cycle SRAM read latency via pre-reads at state transitions.

**Memory subsystem** (`rtl/scratchpad.sv`, `rtl/top.sv`) — Three vector-wide scratchpads:

- SP_A: ROWS × DATA_WIDTH bits per entry (activation rows)
- SP_B: COLS × DATA_WIDTH bits per entry (weight rows)
- SP_C: COLS × ACC_WIDTH bits per entry (result rows)

Single-port synchronous SRAM. External port for host load/readback, muxed with controller access.

### Section 3: Analysis

Interpret the synthesis results.

**Data width scaling** — LUTs scale ~4x per doubling of data width (4.1x INT8→INT16, 3.9x INT16→INT32). Matches theory: multiplier area ∝ product of operand widths. FFs scale exactly 2x (linear in bit width). Takeaway: INT8 is ~4x cheaper than INT16 per MAC — strong incentive for quantized inference.

**Array size scaling** — LUTs scale ~4x per doubling of N (linear in MAC count = N²). Fmax stays roughly constant (106→102 MHz for 2×2→4×4), confirming that the critical path is within a single MAC column, not across the array. Key systolic property: array scales without degrading clock speed.

**Pipeline depth tradeoff** — PD=2 is the sweet spot:

- PD=1→2: Fmax jumps 24% (82→102 MHz) as the critical path splits. LUTs decrease (Yosys optimizes registered multiply better). Throughput peaks at 203 M MACs/s.
- PD=2→3: Fmax barely changes (102→100 MHz) — critical path already short enough for iCE40 fabric. Compute cycles increase (8→9), throughput drops 12%. Extra stage adds 512 FFs for no frequency benefit.
- PD=1 has more LUTs than PD=2/3: Yosys needs more combinational logic for the longer single-cycle path.

### Section 4: Memory Bandwidth & Scale

**Bandwidth model** — During FEED, the array consumes ROWS × DATA_WIDTH bits/cycle from SP_A. Across a full tile:

- SP_A: ROWS × DATA_WIDTH bits/cycle (ROWS cycles)
- SP_B: COLS × DATA_WIDTH bits/cycle (ROWS cycles for weight load)
- SP_C: COLS × ACC_WIDTH bits/cycle (ROWS cycles for writeback, or 2×ROWS-1 for RMW)

For default 4×4 INT8 at 102 MHz: 16 MACs/cycle during FEED, needing 16 bytes/cycle = 1.6 GB/s from SP_A. Total peak scratchpad bandwidth (A+B+C) ~4.8 GB/s. Ratio: ~1 byte/MAC.

Compare to published numbers: NVIDIA H100 provides approximately 2 bytes/FLOP at peak (where FLOP = fused multiply-add = 2 ops, so about 1 byte/MAC equivalent). Cerebras WSE-3, with about 44 GB of distributed on-die SRAM and 900k cores, achieves far higher aggregate on-die bandwidth (200+ bytes/FLOP) by placing memory next to compute — eliminating the off-chip bottleneck entirely. Our design's 1 byte/MAC is in the same ballpark as H100 for a single tile, but wafer-scale architectures show what happens when you remove the memory wall.

**Arithmetic intensity** — Matrix multiply has O(N³) operations on O(N²) data → O(N) arithmetic intensity. Larger tiles are more compute-bound, smaller tiles more memory-bound. Our 4×4 tiles are small; real accelerators use 128×128+ arrays to amortize memory access.

**Wafer-scale thought experiment** (Cerebras WSE-style):

- 4×4 INT8 array: ~4,341 cells on iCE40 ≈ ~0.05 mm² in 7nm
- 46,000 mm² wafer: ~900,000 arrays × 16 MACs = 14.4 billion MACs
- At 102 MHz: ~1.47 ExaMACs/s peak
- SRAM to feed them: 900k × 1.6 GB/s = ~1.4 PB/s aggregate bandwidth
- This is the fundamental challenge of wafer-scale compute: you can fit the compute, but feeding it requires enormous on-die SRAM. Cerebras WSE-3 dedicates ~50% of die area to SRAM for this reason.

### Section 5: Conclusions

Key findings:

- INT8 is ~4x more area-efficient than INT16 per MAC
- PD=2 is the pipeline sweet spot (best throughput per area)
- Memory bandwidth is the bottleneck at scale, not compute

What a real design would change:

- Larger arrays (128×128+) for better arithmetic intensity
- SRAM macros instead of register-based scratchpads
- Double-buffering for latency hiding during tiling
- Wider data buses or multi-bank memory for bandwidth
